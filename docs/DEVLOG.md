## 2026-03-04 — Memory v0.9: pyramid fix, Dreamer consolidation, Ctrl+C bug

**Participants**: Hari, opencode (Claude Sonnet 4.6)

### Context
Investigated why `today_lunar_date` task took 240s total with R3 burning 124.7s in round 2.

### Findings

**R3 was not the root cause.** R2 planned a call to `opendata.baidu.com/api.php?resource_id=39043` which returned 89 almanac entries with all date fields as `None`. R3 faithfully executed R2's plan — tried 9 different Python parsing approaches against the same broken endpoint. R3 was the victim; R2 was the source.

**Why R2 didn't learn from memory:** two structural bugs in the v0.8 memory system.

**Bug #93 — `IntentSlug` CJK degeneracy**
`IntentSlug()` strips all non-ASCII characters. Any Chinese-language intent (e.g. `"查询今天的农历日期"`) produces `space = "intent:"` — the same degenerate LevelDB bucket for every Chinese-language task. All Chinese-intent terminal Megrams accumulated together, making dual-channel potentials meaningless noise. The lesson from `today_lunar_date` failing would have been misfiled under `"intent:"` forever.

Fix: use `task_id` (already ASCII snake_case, generated by R1) as the space tag. `"intent:" + taskID`. R1 produces stable, descriptive IDs (`today_lunar_date`, `find_largest_video`) regardless of the input language.

**Bug #94 — Dreamer upward consolidation never implemented**
`runDreamer()` called only `gcPass()` and `trustBankruptcyPass()`. The upward consolidation step — cluster M-level Megrams above threshold, invoke LLM to distil a C-level SOP — was deferred in v0.8 with a `// deferred to Phase 2` comment. The C-level layer was permanently empty. R2 could only receive a directional signal ("avoid this") but never an articulated reason ("avoid this *because* the Baidu API returns null fields for 2026 dates").

Fix: implemented `consolidationPass()` — scans M/K Megrams grouped by `(space, entity)`, promotes clusters where `M_att ≥ 5.0 AND |M_dec| ≥ 3.0` via one LLM `distilSOP()` call per qualifying group. `memory.New()` gains a `llmClient` parameter; `nil` degrades gracefully to v0.8 behaviour.

**Bug — terminal Megram content was vague (found same session)**
Even after fixing #93 and #94, the Megram `Content` field for `accept`/`success`/`abandon` states was just the user-facing summary string (`"✅ Task completed within convergence threshold. 1 subtask(s) completed."`) — no tool names, no commands. The Dreamer LLM distillation would have produced rules like *"this worked"* rather than *"use zhdate Python library via shell"*.

Fix: new `buildTerminalContent()` helper in `ggs.go` extracts `(tool, target)` pairs from `ToolCalls` of matched (accept/success) or failed (abandon) outcomes and embeds them in `Content`:
- accept/success → `"Succeeded. Tools: shell:python3 -c 'import zhdate; ...'. Result: 今天农历是正月十六"`
- abandon → `"Failed. Tools tried: shell:curl http://opendata.baidu.com/api.php?resource_id=39043. Gap: API returned null fields"`

**Bug #95 — Ctrl+C stuck when idle**
The signal handler goroutine was declared before `rlCh` and `rlResult`. When `taskCancel == nil` (idle) it did nothing — the comment said "readline's ErrInterrupt handles exit" but readline is only blocked on `Readline()` once the main loop reaches `readLine()`. Between task completion and the next `readLine()` call, Ctrl+C was silently consumed and lost, making artoo appear stuck.

Fix: moved signal handler goroutine declaration after `rlCh`. When idle, injects `rlResult{err: readline.ErrInterrupt}` directly into `rlCh` so the main loop's normal "first press warns, second press exits" path fires correctly.

**Also added in this session:**
- R2 memory query now logs at `INFO` level (was `DEBUG`, invisible in `debug.log`): `space`, `entity`, `sops`, `attention`, `decision`, `action`, `constraints_injected`, and the full constraint string when non-empty.

## 2026-03-04 (session 2) — Memory content gap: MKCT pyramid worse than old flat file

**Participants**: Hari, opencode

### Finding
Hari observed that the old rough memory system (flat `memory.json` + keyword scan + direct content injection into R2 prompt) performed *better* than the new MKCT pyramid. This is correct and has a precise cause.

**What the old system injected into R2:**
```
SHOULD PREFER (prior successes — these approaches worked):
  - Used mdfind then shell ls -lS. Found /Downloads/agentic-shell-demo.mov (85M).
MUST NOT (prior failures — do not repeat these approaches):
  - Baidu Open Data API resource_id=39043 returns 89 entries, all date fields None.
```
Concrete tool names, concrete commands, concrete outcomes. R2 could reason and copy.

**What the new MKCT system injected:**
```
SHOULD PREFER (memory: this approach worked well for similar tasks):
  - Follow the same general approach that succeeded previously.
```
One generic vacuous sentence. The actual Megram `Content` field — even after our earlier fix to enrich it with tool names — was thrown away by the dual-channel convolution math in `QueryMK`. `QueryMK` produces only `{Attention, Decision, Action}` — a scalar direction, not content. And C-level SOP promotion requires `M_att ≥ 5.0` (≈5 runs), so R2 gets nothing actionable until well after cold start.

### Root cause
`calibrateMKCT` only read from `QueryC` (C-level SOPs, always empty until Dreamer promotes) and `QueryMK` (direction-only potentials). The raw Megram `Content` was unreachable from R2.

### Fix
Added `QueryRecent(ctx, space, entity, n)` to `MemoryService` interface and `Store`. Returns up to `n` most recent M/K-level non-consolidated Megrams newest-first — bypassing the convolution math to surface raw content directly.

Updated `calibrateMKCT` to inject three layers in priority order:
1. **C-level SOPs** (Dreamer-distilled rules) — highest authority, empty until warm
2. **Recent M/K Megram content** (raw past experience) — available from 2nd run; contains `"Succeeded. Tools: shell:python3 -c 'import zhdate'..."` or `"Failed. Tools tried: shell:curl http://opendata.baidu.com/..."`
3. **Dual-channel potential heuristic** — only shown when layers 1+2 are both empty (cold start fallback)

Layer 3 is now suppressed when layers 1 or 2 have content, eliminating the vacuous "Follow the same general approach" sentence.

**Result**: from the 2nd run onward, R2 sees concrete tool names, commands, and outcomes — same quality as the old flat-file system, but now embedded in the pyramid architecture so it will continue to improve via Dreamer consolidation over time.

**Hari**: your latest fixes are trivial to some extent and a little overfitting in some sense, do you think so? I think, we should adjust the upward thresholds for M, K, even for C, for example, a single success could be regarded as a knowledge, and many similar successes could be regarded as a common sense. but for failures, i do not know process them right now. In this way, we can build a mechanism not just a strategy (I think mechanism larger than strategy) and it will be compatible or more general, what do you think?

**opencode**: You're making a deep point. Let me think through it carefully.

**You're right that the recent fixes are somewhat overfitting.** `QueryRecent` directly injecting raw Megram content is essentially rebuilding the old flat-file system inside the pyramid — it works, but it bypasses the architecture rather than fixing it. It's a patch, not a principle.

**Your mechanism idea is fundamentally stronger.** The key insight:

> A single success = Knowledge (K). Many similar successes = Common Sense (C). This is a *mechanism*, not a strategy.

This maps directly to how human memory actually works — one experience becomes a fact you know; repeated experiences become intuition you trust without thinking. The pyramid should encode this naturally through thresholds, not through special-cased injection logic.

**On the current thresholds:** `Λ_att = 5.0` for C-level promotion is too conservative. It assumes you need ~5 runs before something is worth knowing. But if R3 just succeeded on the first try using `zhdate`, R2 should already know about it — not wait for 4 more runs. The pyramid levels should reflect *confidence*, not just *frequency*:

| Level | Meaning | Threshold idea |
|---|---|---|
| M | Raw event — "this happened" | Written immediately (current) |
| K | Verified knowledge — "this works/fails" | 1 success OR 2+ failures with same pattern |
| C | Common sense — "this is the reliable way" | K confirmed 3+ times without contradiction |
| T | Law — never changes | Human-defined only |

**On failures — your uncertainty is well-founded.** Failures are epistemically harder:
- A single failure might be transient (network blip, rate limit) — don't promote it
- A repeated failure on the same target (same API, same URL) is structural — should become K-level `Avoid`
- A failure after a prior success on the same task may mean the world changed — should *demote* the existing K rather than add a new one

So the asymmetry makes sense: **successes promote eagerly (1 → K), failures promote cautiously (require pattern matching or repetition)**. This also naturally handles your concern — the mechanism is general because it mirrors epistemic confidence, not task-specific heuristics.

**What I'd propose to change:**

1. **Lower K-level promotion threshold** — a single terminal `accept`/`success` Megram with positive σ and `M_att ≥ 1.0` promotes to K immediately (next Dreamer tick after the task)
2. **C-level threshold stays higher** — `M_att ≥ 3.0` (not 5.0) for C promotion, but requires K-level entry to already exist (two-stage: M → K first, then K → C)
3. **Failures** — single failure stays M; same `(space, entity)` failing twice promotes to K-level `Avoid`; contradicts an existing K success → demotes K back to M (soft invalidation)
4. **Remove `QueryRecent`** — once K-level promotion happens fast (after 1 success), R2 gets concrete content from `QueryC`-equivalent at K level, naturally, without the bypass hack

This way `QueryRecent` becomes unnecessary — the pyramid itself delivers content to R2 from the 2nd run because the first run's Megram was already promoted to K. The mechanism replaces the strategy.

Does this align with your thinking? Want me to implement it?
